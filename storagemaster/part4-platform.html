閱讀前請注意

本章的架構概念（三層式、HCI、NAS 整合）適用於各品牌環境。伺服器平台分為兩個獨立的生態：

4.1 三層式架構（3-Tier）#

通用概念

三層式架構（3-Tier Architecture）是傳統企業 IT 基礎設施的標準設計模式，將計算、網路、儲存三個層次明確分離，各自獨立擴展與維護。

三層式的設計優勢

  • 獨立擴展：計算、網路、儲存可以各自擴展，不需要一起升級
  • 故障隔離：儲存設備故障不影響伺服器，伺服器故障不影響儲存資料
  • 靈活搭配：各層可以選擇最適合的廠商和產品
  • 效能可預測：儲存資源不受其他伺服器的計算負載影響

三層式的限制與適用規模

面向說明
初始建置成本需要獨立的 FC Switch、HBA、儲存陣列，前期投資較高
管理複雜度三個層次各有獨立的管理介面，需要跨層次的技術能力
適用規模中大型企業、對效能和可用性有嚴格要求的核心應用，仍是最主流選擇
小規模場景小型企業或非核心應用，HCI 可能是更合適的選擇（見 4.2）

三層式架構本身是通用設計，各品牌的產品都可以填入各層：

4.2 超融合架構（HCI）#

通用概念

HCI 的核心思想

超融合基礎設施（Hyper-Converged Infrastructure，HCI）打破三層式的分離設計，將計算和儲存融合在同一台伺服器節點上。多台節點的本機儲存透過軟體聚合成共享儲存池，提供給所有節點的虛擬機使用。

HCI vs 三層式的取捨

比較項目三層式架構HCI
計算與儲存完全分離，獨立擴展融合在同一節點，必須同步擴展
初始建置成本較高（需 FC 設備）較低（標準 x86 伺服器）
效能儲存效能不受計算負載影響計算和儲存共用 CPU / 網路資源
擴展彈性計算 / 儲存可各自擴展擴展計算必須同時擴展儲存（反之亦然）
適用場景核心應用、高效能需求、大規模環境VDI、中小型環境、邊緣節點、快速部署
管理複雜度三層各有介面，需跨層技能單一管理介面，較簡單

主流 HCI 產品

品牌產品Hypervisor特色
VMware（Broadcom）vSANESXi市占最大，與 vSphere 深度整合
NutanixAOS + AHVAHV（自研）/ ESXi / Hyper-VHCI 發明者，管理介面最友善
DellVxRailESXiDell 硬體 + VMware vSAN，整合認證

IBMStorage Scale（HPC / AI）不限不是傳統 HCI，而是橫向擴展分散式檔案系統，適合 AI 訓練和 HPC

4.3 IBM Power + IBM i / AIX 儲存接法#

IBM 示範
IBM Power 生態說明
IBM Power 是完全獨立於 x86 的伺服器架構，使用 PowerVM 作為 Hypervisor，運行 IBM i 和 AIX 作業系統。這個生態有自己的儲存接法，與 VMware / Linux / Windows 的方式完全不同，不能混淆。

PowerVM 虛擬化下的儲存接法

IBM Power 伺服器透過 PowerVM Hypervisor 提供虛擬化能力。儲存資源從實體 FC HBA 到虛擬機（LPAR）有兩種接法，各有不同的適用場景：

VSCSI vs NPIV 的選擇

比較項目VSCSINPIV
中介層需要 VIOS（Virtual I/O Server）不需要 VIOS，LPAR 直接連 SAN
LPAR 看到的虛擬 SCSI 磁碟（hdisk）虛擬 FC HBA，可直接看到 SAN LUN
FC Zoning以 VIOS 的 WWPN 設定 Zone以各 LPAR 的 vWWPN 設定 Zone，更精細
效能有 VIOS 中介開銷較接近直連效能
適用場景共享 HBA 資源，簡化管理高效能需求、需要精細 Zoning 控制的場景
HA 設計建議雙 VIOS 備援建議雙 HBA + 雙 Fabric

VIOS 的角色

VIOS（Virtual I/O Server）是一個特殊的 LPAR，負責將實體 I/O 資源（FC HBA、Ethernet）虛擬化後提供給其他 LPAR 使用。VIOS 本身不跑業務應用，只做 I/O 仲介。

  • 建議部署雙 VIOS做備援，兩個 VIOS 各連一個 FC Fabric
  • VIOS 故障會影響所有透過它存取儲存的 LPAR，因此 VIOS 的 HA 設計至關重要
  • VIOS 的軟體版本需定期更新，是 IBM Power 環境維護的重要項目

IBM i 和 AIX 的 MPIO 行為

作業系統MPIO 機制特色
AIXAIX MPIO（內建）支援 ALUA，IBM 儲存設備整合最佳，磁碟以 hdisk 命名
IBM i內建多路徑（透明）IBM i 作業系統自動處理多路徑，管理員通常不需要手動設定
跨品牌對照

其他平台的類似虛擬化 I/O 機制

平台類似機制說明
VMware ESXiNPIV（vSphere FC NPIV）VM 可以有自己的 vWWPN，直接對應 SAN Zoning，概念與 IBM NPIV 相同
Linux KVMvirtio-scsi / SR-IOVvirtio-scsi 為虛擬 SCSI 中介；SR-IOV 讓 VM 直接存取實體 HBA，類似 NPIV
Microsoft Hyper-VVirtual FC（vFC）Hyper-V 支援虛擬 FC，VM 可直接連接 SAN，需 HBA 支援 NPIV

4.4 x86 平台儲存整合（VMware / Linux / Windows）#

通用概念

x86 伺服器平台（VMware ESXi、Linux、Windows Server）是目前企業 IT 的主流，可搭配任何品牌的 SAN 儲存設備。以下分三個常見作業系統環境說明儲存整合方式。

VMware vSphere 儲存整合

VMware vSphere 提供三種儲存存取模式，適用不同場景：

模式說明適用場景特色
VMFS（Virtual Machine File System） ESXi 在 SAN LUN 上建立 VMFS 檔案系統，多台 ESXi 共享同一個 VMFS Datastore 一般虛擬機儲存，最常見 多 ESXi 主機可同時存取，支援 vMotion / HA
RDM（Raw Device Mapping） 將 SAN LUN 直接對應給虛擬機，虛擬機看到的是原始磁碟 資料庫（Oracle RAC）、需要直接存取磁碟的應用 效能接近裸機，但管理彈性較低
vVol（Virtual Volumes） 儲存陣列直接管理虛擬機層級的儲存物件，不需要傳統 LUN 支援 vVol 的現代儲存陣列 儲存原生功能（Snapshot / Clone）可直接對應到 VM

VAAI（vStorage APIs for Array Integration）

VAAI 讓 ESXi 可以把特定操作卸載（Offload）到儲存陣列執行，減少 ESXi 主機的 CPU 和網路負載：

  • Full Copy（XCOPY）：VM Clone 時，資料由儲存陣列直接複製，不需要流過 ESXi 主機
  • Block Zeroing：新建 VM 磁碟清零時，由儲存陣列執行，速度大幅提升
  • Atomic Test and Set（ATS）：VMFS 的鎖定機制，改由陣列硬體支援，提升多主機並行存取效率
跨品牌對照

主流儲存品牌均支援 VAAI，以下是各品牌搭配 VMware 的常見整合方式：

Linux 儲存整合

Windows Server 儲存整合

4.5 NAS 整合與 File / Block 共存#

通用概念

什麼時候需要 NAS

Block SAN 是核心應用的主力，但有些場景更適合 NAS（File 儲存）：

  • 多使用者共享檔案：Home Directory、部門共享磁碟機、設計檔案共享
  • 應用原生 NFS/SMB 存取：部分應用（如 VMware Datastore over NFS、容器持久卷）直接使用 NFS
  • 非結構化資料：日誌檔案、備份目標、媒體檔案，不需要 Block 的嚴格 I/O 保證
  • 成本考量：NAS 管理簡單，不需要 FC 基礎設施，適合非關鍵資料

NFS vs SMB 的選擇

協定適用作業系統常見用途效能特性
NFS（Network File System） Linux / Unix / VMware ESXi Linux Home Dir、VMware NFS Datastore、容器 PV 高並行、低延遲，適合大量小檔案
SMB / CIFS Windows（主要）/ macOS / Linux Windows 共享磁碟機、Office 檔案共享 Windows 整合最佳，支援 AD 驗證
IBM 示範

IBM FlashSystem File 功能

IBM FlashSystem 可以在同一台設備上同時提供 Block（SAN）和 File（NAS）服務，不需要額外的 NAS 設備：

  • 支援 NFS v3 / v4、SMB 2.x / 3.x
  • File 和 Block 共用同一個儲存池，統一管理容量
  • File 服務可以配置獨立的網路介面，與 SAN 流量隔離
跨品牌對照

各品牌的 File / Block 統一平台

品牌BlockFile（NAS）統一管理
IBM FlashSystemSAN（FC / iSCSI）NFS / SMB（FlashSystem File）同一設備，統一管理
NetApp ONTAP（AFF）FC / iSCSI / NVMe-oFNFS / SMB / S3ONTAP 統一管理，生態最完整
Dell PowerStoreFC / iSCSI / NVMe-oFNFS / SMB統一管理介面
Everpure FlashArrayFC / iSCSI / NVMe-oFNFS / SMB（FlashArray//C 支援）File 功能較新，仍在成熟中
HPE AlletraFC / iSCSI / NVMe-oFNFS / SMB（部分型號）依型號而異
File 和 Block 共用資源的注意事項
在同一台設備上同時跑 File 和 Block 服務時，兩者共用 CPU、Cache 和網路資源。大量 NAS 存取（如備份流量）可能影響 SAN 的效能。建議為 File 和 Block 服務設定獨立的 QoS 策略，避免互相干擾。
